# Lightdash MCP 设计文档（内部推广版）

本文面向平台维护者、数据团队与 AI 应用负责人，说明 Lightdash MCP 的设计目标、架构、查询优化策略与推广落地方式。

---

## 1. 背景与目标

### 1.1 背景问题

在实际使用中，分析师经常遇到：

- 不知道该调用哪个工具
- 字段 ID 不熟，筛选/排序容易失败
- 一次提问触发多轮试错，响应慢、效率低
- 结果过大，影响上下文和可读性

### 1.2 设计目标

- **业务优先**：先解决“找图表、跑图表、出结果”
- **高容错**：输入尽量宽容，服务端自动归一化
- **少试错**：单次问题尽量 1~3 次调用内完成
- **可治理**：统一认证、日志、限流与错误可观测

---

## 2. 总体架构

```text
业务用户 / 分析师
      ↓
Claude/Cursor（读取 skills 路由）
      ↓
Lightdash MCP（HTTP）
      ↓
Lightdash REST API
```

组件职责：

- `packages/lightdash-skills/`：定义“如何问、怎么路由、失败如何降级”
- `packages/lightdash-mcp/`：提供可调用工具与输入纠错逻辑
- Lightdash API：最终权限控制与数据执行

---

## 3. 工具分层设计

### 3.1 分层原则

- **第一层（业务常用）**：尽量不要求用户理解技术细节
- **第二层（高级查询）**：支持 explore + 维度/指标自定义

### 3.2 当前工具清单

业务优先：

- `lightdash_list_projects`
- `lightdash_search_content`
- `lightdash_list_spaces`
- `lightdash_get_saved_chart`
- `lightdash_run_saved_chart`

高级兜底：

- `lightdash_list_explores`
- `lightdash_get_explore`
- `lightdash_run_metric_query`

---

## 4. 查询优化设计（核心）

本项目已实现“模型友好 + 服务端兜底”的双层优化。

### 4.1 路由层优化（skills）

在 `lightdash-insight-router` 中约束执行策略：

- 优先“已保存图表”路径，减少自定义查询失败率
- 默认小结果集（建议 50~200）
- 单问题限制调用次数（避免无限重试）
- 失败后按规则降级（精简字段 -> 对齐字段 -> 再查询）

### 4.2 服务端优化（MCP）

在 `lightdash_run_metric_query` 中实现：

- 扁平参数接入（降低模型构造复杂度）
- `filters/sorts` 自动归一化
- 字段 ID 自动纠正（支持短名 / 别名映射）
- Explore 元数据缓存（减少重复请求开销）

典型收益：

- `cls_3` 这类业务短字段可自动映射到完整 `fieldId`
- `sorts` 的常见变体可自动转换为 Lightdash 期望结构
- 明显降低 4xx/5xx 触发概率和多轮重试次数

---

## 5. 认证与安全

### 5.1 认证方式

- 推荐 HTTP 请求头：`x-api-key`
- 可按请求覆盖，也支持服务端默认配置（用于开发/托管）

### 5.2 安全原则

- 不在技能或对话中回显密钥
- 日志中密钥脱敏
- 权限完全遵循 Lightdash 项目可见性

---

## 6. 可观测性与故障处理

### 6.1 常见错误分类

- `401/403`：密钥或项目权限问题
- `400`：字段类型/字段 ID 不合法（常见于把维度当指标）
- `422`：查询结构语义不符合接口要求
- `500`：上游执行异常或筛选组合触发的服务端错误

### 6.2 建议排障顺序

1. 确认 `projectUuid` 与时间范围
2. 优先尝试 `run_saved_chart`
3. 自定义查询时先减维度、减 limit、减筛选复杂度
4. 仍失败再 `get_explore` 对齐字段

---

## 7. 推广落地建议

### 7.1 推广对象分层

- 分析师/业务：只培训“怎么提问、怎么看结果”
- 数据团队：补充高级查询与字段口径说明
- 平台团队：维护配置、日志、版本升级

### 7.2 培训话术（建议）

- “先找图表，找不到再做自定义”
- “先问结论，再下钻细节”
- “一次先查小范围，确认后再扩大”

### 7.3 KPI（可量化）

- 单问题平均调用次数
- 首次成功率（一次查询成功占比）
- 人均问答完成时长
- 失败类型分布（401/400/422/500）

### 7.4 运行门禁（必须启用）

- 缺少最小信息集（项目、时间、指标）时禁止执行查询
- 单问题最多 3 次工具调用，超过则返回阶段性结果与补充问题；**类目校验/单层降级**见 `lightdash-mcp-query-sop.md` 2.1，允许最多 **4** 次
- 连续两次失败必须停止重试并输出失败回报模板

## 8. 版本建议与后续演进

短期：

- 增加失败时“归一化后 query 片段”调试输出（脱敏）
- 增强筛选语义兼容（更多 operator/值结构）

中期：

- 建立常见业务问题模板库（按部门）
- 增加基于问题意图的自动字段推荐

长期：

- 引入查询质量评分与自动重写建议
- 建立跨项目的统一语义层字典
